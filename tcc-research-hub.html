WebGL

WebGL e o Pipeline de Renderização Clássico

A API gráfica da web baseada em OpenGL ES 2.0/3.0. Compreender seu pipeline é fundamental para medir seus gargalos.

O que é: WebGL (Web Graphics Library) é uma API JavaScript que permite renderização de gráficos 2D e 3D acelerados por GPU diretamente no navegador, sem necessidade de plugins. É baseada no padrão OpenGL ES 2.0 (WebGL1) e 3.0 (WebGL2).


Pipeline: Vertex Shader → Rasterização → Fragment Shader → Framebuffer. O CPU precisa emitir cada draw call individualmente, criando um gargalo de comunicação CPU-GPU no estado atual da API.


Limitações críticas para seu TCC: (1) Cada draw call tem overhead de validação de estado no driver; (2) Sem suporte nativo a compute shaders (WebGL2 limitado); (3) O modelo de threading é single-threaded; (4) Instanced drawing existe mas com limitações de extensão.


Relevância para o TCC: Você precisa documentar QUANTOS draw calls um modelo arquitetônico de alta densidade gera e COMO isso afeta a CPU. Essa é a métrica central que justifica a necessidade do WebGPU.

WebGPU

WebGPU: A Nova Geração de Gráficos na Web

API moderna inspirada em Vulkan, Metal e D3D12. Reduz drasticamente o overhead da CPU e habilita compute shaders.

O que é: WebGPU é o sucessor do WebGL, padronizado pelo W3C com implementações no Chrome 113+ (estável) e Firefox/Safari em desenvolvimento. É inspirado em Vulkan, Metal (Apple) e Direct3D 12.


Avanços arquiteturais: (1) Command Buffers — comandos de renderização são gravados fora da thread principal e enviados em lote; (2) Compute Pipelines — shaders de computação gerais na GPU; (3) Bind Groups — gerenciamento explícito de recursos mais eficiente; (4) Render Bundles — pre-gravação de sequências de render.


WGSL: WebGPU Shading Language é a nova linguagem de shader, substituindo GLSL. Tipagem mais estrita e validação em tempo de compilação.


Relevância para o TCC: O render bundle é o recurso mais crítico para seu experimento: ele permite "gravar" uma sequência de draw calls e reexecutá-la com overhead mínimo, atacando diretamente o gargalo que você está medindo.

Draco

Compressão de Malhas com Draco

Algoritmo de compressão geométrica do Google. Reduz dramaticamente o tamanho de arquivos .glTF/.GLB sem perda visual significativa.

O que é: Draco é uma biblioteca open-source do Google para compressão e descompressão de malhas 3D e nuvens de pontos. Integrada nativamente ao formato glTF 2.0 como extensão KHR_draco_mesh_compression.


Como funciona: Usa EdgeBreaker para compressão de conectividade da malha e quantização de atributos (posição, normais, UVs). Uma malha de 50MB pode ser reduzida para 5-8MB.


Trade-off importante: A descompressão ocorre na CPU em tempo de carregamento. Para seu TCC, você precisa medir o TTFF (Time to First Frame) e separar o tempo de descompressão Draco do tempo de upload para GPU.


Three.js Integration: DRACOLoader + GLTFLoader. O decoder Draco pode rodar em WebWorker para não bloquear a thread principal — isso é relevante para sua metodologia.

LOD

Level of Detail (LOD) Dinâmico

Técnica fundamental de otimização: renderizar versões menos detalhadas de objetos distantes, reduzindo o custo de vértices.

O que é: LOD é uma técnica onde múltiplas versões de uma malha (alta, média, baixa resolução) são preparadas antecipadamente. O sistema seleciona qual versão renderizar baseado na distância à câmera ou no tamanho em tela.


Tipos: (1) Discrete LOD — switch abrupto entre versões (Three.js LOD class); (2) Continuous LOD — simplificação progressiva em tempo real (complexo); (3) Impostors/Billboard LOD — substituição por imagem 2D a longa distância.


Geração de LODs: Você pode usar Meshoptimizer ou o gltf-transform CLI para gerar automaticamente múltiplos níveis de LOD a partir do modelo de alta fidelidade.


Para o TCC: Medir o impacto do LOD isoladamente vs o impacto do WebGPU isoladamente, e depois a combinação das duas técnicas, é uma estrutura experimental sólida e publicável. Define grupos de controle claros.

Draw Calls

Gargalo de Draw Calls e Batching

O problema central que motiva seu TCC: por que muitos draw calls destroem o desempenho e como o WebGPU resolve isso.

O problema: Cada draw call em WebGL requer que o driver valide o estado completo da GPU (shaders ativos, buffers vinculados, texturas, etc.) antes de executar. Em modelos arquitetônicos com milhares de objetos, isso gera dezenas de milhares de draw calls por frame.


Custo real: Um draw call em WebGL custa entre 5-50 microsegundos de CPU. A 60 FPS você tem ~16ms por frame. Com 10.000 draw calls × 10μs = 100ms só de overhead de submissão — 6× acima do budget de frame.


Soluções WebGL: Instanced Drawing (drawElementsInstanced), Geometry Batching manual, Texture Atlases, Merged Geometries. Todas com limitações e custo de implementação.


Solução WebGPU: Render Bundles gravam os draw calls uma vez e os reexecutam sem revalidação. O overhead por draw call cai drasticamente para objetos estáticos — exatamente o caso de maquetes arquitetônicas.

BIM & Arquitetura 3D

Modelos BIM e Alta Densidade Poligonal

O que define um modelo arquitetônico como "complexo" e por que BIM é o caso de uso ideal para este benchmark.

BIM (Building Information Modeling): Modelos que integram geometria 3D com metadados (materiais, custos, cronogramas). Formatos: IFC, RVT (Revit), que são exportados para glTF para visualização web.


Características que estressam renderizadores: (1) Hierarquia profunda de objetos (cada parede, janela, porta é um mesh separado); (2) Materiais únicos por objeto (impossibilita batching simples); (3) Contagem de polígonos: modelos reais variam de 500k a 10M+ triângulos; (4) Oclusão complexa (objetos dentro de objetos).


Datasets públicos recomendados: (1) Revit Sample Project — exportável em glTF; (2) IFC Open Shell — modelos de teste padronizados; (3) Sketchfab — modelos arquitetônicos com licença CC; (4) Gerar proceduralmente com Blender para controle total da densidade.


Para o TCC: Definir claramente o modelo de controle (fixo, documentado, reprodutível) é o que torna seu experimento cientificamente válido.

Three.js WebGPURenderer

Three.js como Abstração Dual (WebGL + WebGPU)

Como o Three.js implementa suporte aos dois backends e o que você precisa saber para seu protótipo.

Arquitetura: Three.js separa a lógica de cena (Scene Graph) do backend de renderização. O WebGLRenderer é estável; o WebGPURenderer (via pacote three/addons) está em desenvolvimento ativo desde a versão r150+.


Node System (TSL): O Three.js WebGPU usa um sistema de node materials onde shaders são construídos como grafos. Isso é a base do novo sistema de materiais. Para seu TCC, use os materiais equivalentes: MeshStandardNodeMaterial ao invés de MeshStandardMaterial.


Ponto crítico: Nem todas as features do WebGLRenderer têm equivalente no WebGPURenderer ainda. Isso pode introduzir variáveis de confusão nos seus resultados. Documente quais features você usa e confirme o suporte em ambos os backends.


Alternativa: Implementar os dois renderers diretamente (sem Three.js) dá mais controle mas eleva muito a complexidade. Para um TCC, Three.js é a escolha correta e justificável.

GPU Memory

VRAM, Texturas e Gerenciamento de Memória GPU

Como alocar, monitorar e otimizar o consumo de memória de vídeo em ambientes de alta densidade.

Hierarquia de memória: CPU RAM → VRAM (rápida, acesso direto pela GPU) → Cache de textura (L1/L2 na GPU). Em WebGL, você não tem controle explícito sobre essa hierarquia. Em WebGPU, o gerenciamento é mais explícito.


Problemas comuns: (1) Texturas não comprimidas desperdiçam VRAM (use KTX2/Basis Universal); (2) Geometrias duplicadas na GPU por falta de instancing; (3) Render targets desnecessários; (4) Vazamento de recursos (geometrias/materiais não descartados).


Monitoramento: Em WebGL: WEBGL_debug_renderer_info + extensão EXT_disjoint_timer_query. Em WebGPU: GPUCommandEncoder.writeTimestamp (experimental) + Chrome DevTools GPU profiler.


Para o TCC: Definir exatamente COMO você vai medir VRAM é metodologicamente importante. A API performance.memory mede heap JS, não VRAM. Use o Chrome Task Manager ou as APIs de profiling do browser.

Guia de Implementação do Protótipo
Estrutura técnica recomendada para construir o ambiente experimental com Vite + Three.js.
⚠️ Atenção: O WebGPURenderer do Three.js ainda está em evolução. Use a versão r167+ e verifique o CHANGELOG antes de cada atualização. Teste no Chrome 113+ (WebGPU estável).
🏗️

Estrutura do Projeto

  • 01Inicialize com Vite (npm create vite@latest tcc-benchmark) — bundler rápido com HMR e suporte nativo a ES modules
  • 02Instale Three.js versão r167+ (npm install three@latest)
  • 03Separe em módulos: renderer-webgl.js, renderer-webgpu.js, scene-builder.js, metrics-collector.js, camera-path.js
  • 04Use a mesma scene graph para ambos os renderers — isso garante que a diferença medida é apenas o backend gráfico
  • 05Implemente um ?renderer=webgl|webgpu query param para trocar o backend via URL
📦

Setup do Modelo 3D

  • 01Escolha um modelo glTF de domínio público com 500k–2M triângulos (Sketchfab / KhronosGroup samples)
  • 02Use gltf-transform CLI para aplicar Draco e gerar LODs automaticamente
  • 03Gere 3 versões do mesmo modelo: sem otimização, com Draco, com Draco + LOD
  • 04Documente os metadados de cada versão: nº de triângulos, nº de meshes, nº de materiais, tamanho em disco
  • 05Configure o DRACOLoader para usar WebWorker e não bloquear a thread principal
📸

Trajetória de Câmera Automatizada

  • 01Grave waypoints manualmente ao navegar pelo modelo pela primeira vez
  • 02Use CatmullRomCurve3 do Three.js para interpolar suavemente entre waypoints
  • 03A trajetória deve ter duração fixa (ex: 60 segundos) e ser idêntica em ambos os renderers
  • 04Inclua cenas de stress: visão de toda a edificação, zoom em geometria densa, transições rápidas
  • 05Salve os waypoints em JSON para garantir reprodutibilidade dos testes
📊

Coleta de Métricas

  • 01Instale stats.js para monitorar FPS e Frame Time em tempo real com overlay visual
  • 02Use performance.now() para medir TTFF (tempo do primeiro frame renderizado após load)
  • 03Implemente um array de log: a cada frame, armazene {timestamp, fps, frameTime}
  • 04Ao final do teste, exporte os dados como CSV via Blob + URL.createObjectURL
  • 05Para VRAM: use Chrome DevTools → Performance → Memory tab durante a gravação

// main.js — seleção de renderer via query param
import { createWebGLScene } from './renderer-webgl.js';
import { createWebGPUScene } from './renderer-webgpu.js';

const params = new URLSearchParams(window.location.search);
const backend = params.get('renderer') ?? 'webgl';

const TTFF_start = performance.now();

if (backend === 'webgpu') {
  const supported = await navigator.gpu?.requestAdapter();
  if (!supported) { document.body.textContent = 'WebGPU não suportado'; return; }
  await createWebGPUScene(TTFF_start);
} else {
  await createWebGLScene(TTFF_start);
}

// metrics-collector.js — coleta frame-a-frame
export class MetricsCollector {
  constructor() {
    this.data = [];
    this.lastTime = performance.now();
    this.frameCount = 0;
  }

  tick() {
    const now = performance.now();
    const frameTime = now - this.lastTime;
    this.data.push({ t: now, ft: frameTime, fps: 1000 / frameTime });
    this.lastTime = now;
    this.frameCount++;
  }

  exportCSV(filename) {
    const csv = 'timestamp,frameTime,fps\n' +
      this.data.map(r => `${r.t},${r.ft},${r.fps}`).join('\n');
    const blob = new Blob([csv], { type: 'text/csv' });
    const a = document.createElement('a');
    a.href = URL.createObjectURL(blob);
    a.download = filename; a.click();
  }
}
